# Chocolatine V2 — Plan d'architecture

## Objectif

V2 a un seul objectif structurant : **support multi-dialectes SQL** (MySQL, PostgreSQL, SQLite, SQL Server).

Cela implique un refactor interne complet — remplacer choc-expr par des méthodes `build()` explicites et introduire un système de renderers. **L'API publique ne change pas** : `Query`, `Col`, `With`, `Window`, etc. restent identiques pour les utilisateurs.

---

## Problème actuel

choc-expr embarque le SQL directement dans des strings de template :

```python
super().__init__(
    "CREATE TABLE {_table}~(@{_auto_id}:id MEDIUMINT NOT NULL AUTO_INCREMENT, :;...)"
)
```

Il n'y a nulle part où accrocher une logique de dialecte. Le SQL MySQL est coulé dans le béton.

Problèmes secondaires :
- Noms d'attributs internes réservés non documentés (ex. `_expr` casse silencieusement)
- Le comportement de `compact` comme attribut d'instance shadow les properties des sous-classes
- Templates complexes illisibles (ex. `@{_using}:USING {$(_condition)}:ON {_condition};`)

---

## Solution : Renderer pattern

Chaque classe garde sa logique de structure (quels champs, quel ordre), mais délègue le rendu des fragments dialecte-spécifiques à un **renderer**.

### Interface du renderer

```python
class SqlRenderer:
    # Types
    def type_string(self) -> str: ...
    def type_integer(self) -> str: ...
    def type_bigint(self) -> str: ...
    def type_float(self) -> str: ...
    def type_boolean(self) -> str: ...
    def type_datetime(self) -> str: ...
    def type_date(self) -> str: ...
    def type_text(self) -> str: ...

    # DDL
    def auto_increment_col(self) -> str: ...       # "id MEDIUMINT NOT NULL AUTO_INCREMENT"
    def primary_key_clause(self) -> str: ...        # "PRIMARY KEY (id)"

    # Pagination
    def limit_offset(self, limit, offset) -> str: ...  # "LIMIT n OFFSET m" vs "TOP n"

    # Fonctions dialecte-spécifiques
    def fn_if(self, cond, a, b) -> str: ...        # IF() vs IIF() vs CASE WHEN
    def fn_ifnull(self, a, b) -> str: ...          # IFNULL() vs COALESCE() vs ISNULL()
    def fn_date_format(self, col, fmt) -> str: ...  # DATE_FORMAT vs TO_CHAR vs strftime
    def on_conflict(self, assignations) -> str: ... # ON DUPLICATE KEY UPDATE vs ON CONFLICT
```

### Renderers par dialecte

```python
class MySQLRenderer(SqlRenderer):
    def type_string(self): return "VARCHAR(255)"
    def auto_increment_col(self): return "id MEDIUMINT NOT NULL AUTO_INCREMENT"
    def fn_if(self, cond, a, b): return f"IF({cond}, {a}, {b})"
    def fn_ifnull(self, a, b): return f"IFNULL({a}, {b})"
    def fn_date_format(self, col, fmt): return f"DATE_FORMAT({col}, {fmt})"
    def on_conflict(self, assignations): return f"ON DUPLICATE KEY UPDATE {assignations}"
    def limit_offset(self, limit, offset): ...

class PostgreSQLRenderer(SqlRenderer):
    def type_string(self): return "TEXT"
    def auto_increment_col(self): return "id SERIAL"
    def fn_if(self, cond, a, b): return f"CASE WHEN {cond} THEN {a} ELSE {b} END"
    def fn_ifnull(self, a, b): return f"COALESCE({a}, {b})"
    def fn_date_format(self, col, fmt): return f"TO_CHAR({col}, {fmt})"
    def on_conflict(self, assignations): return f"ON CONFLICT DO UPDATE SET {assignations}"
    def limit_offset(self, limit, offset): ...

class SQLiteRenderer(SqlRenderer):
    def type_string(self): return "TEXT"
    def auto_increment_col(self): return "id INTEGER PRIMARY KEY AUTOINCREMENT"
    def fn_if(self, cond, a, b): return f"CASE WHEN {cond} THEN {a} ELSE {b} END"
    def fn_ifnull(self, a, b): return f"IFNULL({a}, {b})"
    def fn_date_format(self, col, fmt): return f"strftime({fmt}, {col})"
    def on_conflict(self, assignations): return f"ON CONFLICT DO UPDATE SET {assignations}"
    def limit_offset(self, limit, offset): ...

class SQLServerRenderer(SqlRenderer):
    def type_string(self): return "NVARCHAR(255)"
    def auto_increment_col(self): return "id INT IDENTITY(1,1)"
    def fn_if(self, cond, a, b): return f"IIF({cond}, {a}, {b})"
    def fn_ifnull(self, a, b): return f"ISNULL({a}, {b})"
    def fn_date_format(self, col, fmt): return f"FORMAT({col}, {fmt})"
    def on_conflict(self, assignations): return f"MERGE ..."  # complexe, à traiter séparément
    def limit_offset(self, limit, offset): ...  # TOP n dans SELECT, pas en fin de requête
```

### Signature de build()

```python
# Renderer MySQL par défaut — aucun changement pour les utilisateurs existants
query.build()                              # MySQL (défaut)
query.build(renderer=PostgreSQLRenderer()) # PostgreSQL
query.build(renderer=SQLiteRenderer())     # SQLite
```

Le renderer se propage en cascade : `Query.build(r)` appelle `SelectFrom.build(r)`, `Where.build(r)`, etc.

## Refactor interne : choc-expr → build() explicite

Chaque classe remplace son appel à `super().__init__(template)` par une méthode `build(renderer)`.

**Avant (choc-expr) :**
```python
class Col(ChocExpr):
    def __init__(self, ...):
        super().__init__("{full_name}@{_alias}: AS {_alias}:;")
```

**Après :**
```python
class Col:
    def build(self, renderer=MySQLRenderer()) -> str:
        name = self.full_name
        return f"{name} AS {self._alias}" if self._alias else name
```

**Avant (choc-expr) :**
```python
class Join(ChocExpr):
    def __init__(self, ...):
        super().__init__(
            "{_join_type.value} JOIN {_table~}@{_using}:USING {$(_condition)}:ON {_condition};"
        )
```

**Après :**
```python
class Join:
    def build(self, renderer=MySQLRenderer()) -> str:
        sep = " " if self._compact else "\n"
        table = self._table.build(renderer)
        if self._condition is None:
            return f"{self._join_type.value} JOIN {table}"
        if self._using:
            cond = f"USING ({', '.join(self._condition)})"
        else:
            cond = f"ON {self._condition.build(renderer)}"
        return f"{self._join_type.value} JOIN {table}{sep}{cond}"
```

## Différences dialectes à gérer

| Fonctionnalité | MySQL | PostgreSQL | SQLite | SQL Server |
|---|---|---|---|---|
| Auto-increment | `MEDIUMINT AUTO_INCREMENT` | `SERIAL` | `INTEGER AUTOINCREMENT` | `INT IDENTITY(1,1)` |
| String type | `VARCHAR(255)` | `TEXT` | `TEXT` | `NVARCHAR(255)` |
| LIMIT/OFFSET | `LIMIT n OFFSET m` | `LIMIT n OFFSET m` | `LIMIT n OFFSET m` | `OFFSET m ROWS FETCH NEXT n ROWS ONLY` |
| `IF(cond, a, b)` | `IF()` | `CASE WHEN` | `CASE WHEN` | `IIF()` |
| `IFNULL(a, b)` | `IFNULL()` | `COALESCE()` | `IFNULL()` | `ISNULL()` |
| `DATE_FORMAT` | `DATE_FORMAT(d, fmt)` | `TO_CHAR(d, fmt)` | `strftime(fmt, d)` | `FORMAT(d, fmt)` |
| `ON DUPLICATE KEY` | oui | `ON CONFLICT DO UPDATE` | `ON CONFLICT DO UPDATE` | `MERGE` (complexe) |
| `DATEDIFF(a, b)` | nb de jours `a - b` | `(a - b)` (interval) | `julianday(a) - julianday(b)` | `DATEDIFF(day, b, a)` (ordre inversé) |
| `INTERSECT` | depuis MySQL 8.0 | oui | oui | oui |

## Ce qui ne change pas

- Toute l'API publique : `Query`, `Col`, `With`, `Window`, `Subquery`, `Coalesce`, etc.
- Les opérateurs Python : `==`, `>`, `&`, `|`, `~`, `<<`, `>>`
- Les méthodes fluentes : `.alias()`, `.sum()`, `.filter()`, `.join()`, etc.
- `compact=True/False`

## Étapes de migration

### Phase 1 — Supprimer choc-expr, réécrire les build()
- Remplacer `ChocExpr` par une classe de base simple avec `build(renderer)` abstraite
- Réécrire classe par classe en commençant par les feuilles (`Col`, `Table`, `Condition`)
- Garder les tests existants comme filet de sécurité (ils valident le comportement MySQL)

### Phase 2 — Implémenter MySQLRenderer
- Extraire toutes les strings MySQL hardcodées vers `MySQLRenderer`
- `build()` sans argument reste équivalent à `build(MySQLRenderer())`
- Tous les tests passent sans modification

### Phase 3 — Implémenter PostgreSQLRenderer
- Ajouter un suite de tests `tests/dialects/test_postgresql.py`
- Couvrir les cas de divergence un par un

### Phase 4 — SQLite, SQL Server
- Même approche

## Complexités connues

**SQL Server LIMIT** : `TOP n` va dans le `SELECT`, pas en fin de requête. Ça implique que `SelectFrom.build()` doit recevoir la limite pour l'injecter au bon endroit si le renderer est SQL Server. C'est le cas le plus architecturalement complexe.

**`ON DUPLICATE KEY` → `MERGE` (SQL Server)** : `MERGE` est une syntaxe complète, pas juste un suffixe. À traiter comme un mode de query à part entière plutôt qu'une transformation du renderer.

**`date_format` SQLite** : l'ordre des arguments est inversé (`strftime(fmt, date)` vs `DATE_FORMAT(date, fmt)`). Le renderer doit gérer l'inversion, pas la classe appelante.
